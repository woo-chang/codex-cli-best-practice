# 커밋 규칙

이 프로젝트의 번역 관련 커밋 메시지는 한글로 작성합니다. 접두사는 영어로 유지하고, 요약과 본문은 한글로 작성합니다.

## 커밋 메시지 형식

```text
<영어 접두사>: <한글 변경 요약>

<본문 (선택)>
```

## 접두사

| 접두사 | 용도 |
|---|---|
| `feat` | 새 하네스 파일, 새 규칙, 새 기능 추가 |
| `translate` | 문서 번역 작업 |
| `update` | 기존 규칙, 문서, 보고서 갱신 |
| `fix` | 번역 오류, 링크 오류, 구조 오류 수정 |
| `sync` | upstream 동기화 및 재번역 대상 갱신 |
| `verify` | 역번역 검증, 구조 검증, 리뷰 기록 |
| `config` | 설정 파일 또는 운영 정책 조정 |

## 메시지 규칙

- 접두사는 영어로 유지합니다.
- 요약은 한글로 간결하게 작성합니다.
- 여러 파일을 묶는 경우 본문에 범위나 파일군을 적습니다.
- 기술 식별자, 파일 경로, 도구 이름은 영어 그대로 둡니다.

## 커밋 단위

- 번역 하네스 구조 추가는 한 번에 커밋할 수 있습니다.
- 번역 작업은 가능한 한 배치 단위로 커밋합니다.
- upstream 동기화는 번역과 섞지 않고 별도 커밋으로 남깁니다.
- 역번역 검증 후 수정은 번역 커밋과 분리하는 편이 좋습니다.

## 커밋 전 검증

커밋 전에 다음을 확인합니다.

1. 현재 브랜치가 `main` 또는 `master`가 아닌지 확인합니다.
2. 민감 파일 (`.env`, `*.pem`, `*.key`, `*token*`, `*secret*`)이 staging되지 않았는지 확인합니다.
3. 번역 대상이 아닌 바이너리 파일 (`.png`, `.jpg`, `.gif`, `.webp`, `.svg`, `.mp3`, `.wav`) 수정이 의도된 것인지 확인합니다.
4. 스테이징된 마크다운 파일에서 코드 펜스와 인라인 코드가 번역으로 깨지지 않았는지 확인합니다.
5. 프론트매터와 TOML/JSON/YAML 키가 영어로 유지되는지 확인합니다.
6. 상대 링크와 이미지 경로가 깨지지 않았는지 확인합니다.
7. 번역 커밋이라면 필요한 경우 `.originals/`, `reports/`, 역번역 검증 결과가 함께 갱신됐는지 확인합니다.

## 금지 사항

- `main`이나 `master`에 직접 커밋하지 않습니다.
- `git commit --amend`는 사용하지 않습니다.
- `git push --force`는 사용하지 않습니다.
- `--no-verify`로 검증을 우회하지 않습니다.
- upstream 동기화 커밋과 번역 커밋을 한 커밋에 섞지 않습니다.
